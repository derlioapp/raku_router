## Why raku_router

Every router pushes navigation complexity *somewhere* — `go_router` into
redirects/shells, `auto_route` into a build step, Navigator 2.0 onto you.
raku_router's bet: for most apps the real need is modest, and a small reactive core
beats a big framework.

<CardGrid>
	<Card title="Type-safe, no codegen" icon="seti:dart">
		Routes are plain `sealed` classes; an exhaustive `switch` is your route
		table. No `build_runner`, no `.gr.dart`, no annotations.
	</Card>
	<Card title="Nested tabs built in" icon="list-format">
		Each tab keeps its own persistent back stack — what `StatefulShellRoute`
		does, but by default.
	</Card>
	<Card title="Deep linking from a tree" icon="puzzle">
		A URL's *structure* reconstructs the navigation stack, both ways — and
		browser back/forward preserves state.
	</Card>
	<Card title="No state / design-system dep" icon="approve-check">
		The core only needs `flutter`. Drops into Material, Cupertino, or a custom
		design system equally. WASM-ready.
	</Card>
</CardGrid>
